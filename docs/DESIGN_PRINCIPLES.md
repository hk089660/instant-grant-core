# 設計思想と次フェーズ

この文書は、`instant-grant-core` の現状の位置づけと、その先に置いている設計思想を明文化するためのものです。現行 repository にある実装だけでなく、「なぜ今の形なのか」「次に何を壊し、何を置き換えるのか」まで含めて説明します。

## 1. 現在地

現行システムは **Phase 1: 機能検証の PoC** です。

ここで検証しているのは主に次です。

- UX と監査証跡を両立できるか
- L1 / L2 の連動で、参加証跡と受給証跡を結び付けられるか
- PoP と hash chain を使って、第三者検証可能な運用導線を作れるか
- 現場運用に耐える管理 UI / 利用者 UI を短期間で成立させられるか

このフェーズでは、最速で仮説検証することを優先しています。そのため trust model はまだ中央集約寄りで、Worker / PoP signer / 運用者に依存する前提が残っています。

## 2. なぜ今は PoC なのか

この repository の初期実装は、短期間で L1 / L2 連動の成立性を検証するために組まれました。目的は「完成品を宣言すること」ではなく、「本当に作る価値があるのか」「どこが技術的ボトルネックか」を切り出すことでした。

したがって、現状のコードベースは次の意味で意図的に PoC です。

- Web2 的な中央集約境界をまだ含む
- 単一 signer や静的運用導線など、将来置き換えるべき部分が明示的に残っている
- 先に検証すべき UX / 監査 / L1 / L2 接続を優先し、完全な trustless 化は次フェーズへ送っている

## 3. 開発方針

このプロジェクトは AI を使って高速に PoC を組み立てていますが、AI 出力をそのまま信頼する方針ではありません。採用している前提は次です。

- AI は実装速度を上げる補助であり、設計責任者ではない
- AI が出したコードは人間が一通り読む
- AI 自身にもレビューとメタ認知をさせる
- 実装を試し、ビルド、テスト、挙動確認を通ったものだけを残す

狙いは、AI の速度を使いながらも、ハルシネーションや局所最適化で壊れた設計を repo に定着させないことです。

## 4. Phase 2 の設計思想

目標は **「攻撃を検知して自壊し、自律的に復旧するセキュリティ基盤」** です。

この設計思想は、生物学的な防御反応を計算機システムへ写像する発想に近いものです。守りたい原則は次の 4 つです。

- 検知する: 侵害兆候や異常を早く見つける
- 隔離する: 被害範囲を局所化する
- 零化する: 長寿命の攻撃面を減らし、必要なら鍵やセッションを破棄する
- 復旧する: 証跡を壊さず、正系へ戻す

この考え方は、単なる「防ぐ」ではなく、侵害を前提に動的に形を変え続ける `Moving Target Defense` と、`zeroization / key rotation / self-healing` を組み合わせた基盤を目指すものです。

## 5. 現在の主要な未解決課題

### 5.1 単一 signer / 中央集約境界

現在は `api-worker` 側の PoP signer と運用制御が、強い中央集約境界になっています。特に「エッジサーバが単一の長寿命鍵を持つ」構造は、最重要の縮小対象です。

### 5.2 静的 QR / 物理空間との接点

会場導線で静的な QR や印刷物を入口にすると、盗撮、再利用、後追い侵入の余地が残ります。これは暗号プロトコル以前に、物理層との接点として弱い部分です。

## 6. レイヤ別の実装方針

### 6.1 ネットワーク / 制御層

単一鍵依存を減らすため、次を優先候補にします。

- TEE / secure enclave を使った署名環境
- エフェメラル鍵のラチェット構造
- 役割鍵の分離
- `threshold signer` や `multisig` の導入
- fail-open ではなく fail-closed を基本にした停止 / 切替 / 復旧

狙いは、運営者ですら長寿命の秘密鍵をそのまま持ち続けない構造へ移すことです。

### 6.2 物理 / 会場層

静的 QR を減らすため、次の導線を検討します。

- 会場タブレットなどでの動的 QR 表示
- TOTP など時間ベースのワンタイムコード
- 数秒単位での失効と更新
- 盗撮されてもすぐ無効化される短寿命トークン

狙いは、紙や静的 URL のコピーだけでは侵入に使えない設計へ変えることです。

### 6.3 監査 / 復旧層

次フェーズでは、証跡を壊さずに復旧できることを重視します。

- hash chain と immutable audit sink を継続する
- 異常時は cutover / isolate / rotate を優先する
- in-place 修正よりも、証跡保全を優先した切替を標準手順にする
- 復旧ランブックをコードと運用の両方で持つ

## 7. repository への読み替え

現行 repo の位置づけは次のとおりです。

| 領域 | 現在 | 次フェーズ |
| --- | --- | --- |
| `grant_program` | claim / receipt / PoP 検証の基礎層 | より強い role separation、閾値署名、復旧運用との整合 |
| `api-worker` | 中央集約境界を含む PoP / audit / event 運用層 | enclave / ratchet / dynamic trust minimization への移行対象 |
| `wene-mobile` | 現場 UX と claim 導線の検証 | 動的 QR、短寿命セッション、強化された現場認証導線 |
